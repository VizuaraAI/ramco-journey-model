01 · The problem

An ERP has hundreds of screens.
A user has one question.

Ramco's Purchase Order module alone exposes 14 user-facing journeys, 222 UI tasks, 1,219 stored procedures, and dozens of conditional sub-flows. A user wants to type one sentence: "create a purchase order for SUP-100 in USD."

Traditional chatbots try to bridge that gap by mining training documents. We tried — and it failed because documents are derivative: they describe what the system does, but they're authored inconsistently and frequently out of sync with the screens. A manual might say "Create Direct PO" using prose that says nothing about which SP fires or which table gets written. And critically, documents tell you nothing about the conditional logic — that PO type = Capital triggers extra required slots is buried in a stored procedure, not on page 47 of a manual.

So we threw the documents away and went directly to the source.

02 · The six artifacts

Six files. Six roles.
Together: everything the bot needs.

Ramco ships six kinds of artifact per module. Each answers a different question. Click Show file on any layer to see the real content with the meaningful lines highlighted.

1

The module manifest

PO/ScreenObjects/PO/PO_info.xml  ·  1,518 lines

The whole module's table of contents — every Activity → ILBO → Page → Task. Tasks are typed: FETCH (load), INIT (initialise dropdowns), TRANS (commit / "Save"), LINK (open sub-screen), UI (validate a field), HELP (popup search).

2

The screen state machine

PO/ScreenObjects/PO/PoCrt_PoCrtMain_State.xml  ·  243 lines

One per main screen. Declares the services on that screen (e.g. PoCrtMain has three: Fetch, Create, Approve) and the named visibility states (e.g. project_details_on). State IDs name conditions under which controls show/hide — these are state-machine splices.

3

The form template

PO/ScreenObjects/PO/Pocrt_pocrtmain.htm  ·  1,790 lines

The screen's HTML layout. At the top is a <meta name="Tasks"> tag enumerating every task on this screen with its description and type. Below: the actual form sections with field labels, types, and datatypes. This is where we learn what slots exist.

4

The screen behaviour (JS)

PO/ScreenObjects/PO/Pocrt_pocrtmain_user.js  ·  227 lines

One switch statement with the success message per task. Useful as a cross-check when the manifest description is terse.

5

The service catalog

PO/ModelInfo/Service_details_PO.csv  ·  6,778 rows

The map: (activity, screen, task) → stored procedure call chain. Ordered by sequenceno. When the user clicks "Save" on PoCrtMain, this CSV tells us exactly which SPs run in what order.

6

The stored procedures

PO/SPS/pocrmn_sp_crt_hdrchk.sql  ·  and 1,218 others

The actual business logic. Inside each SP: parameter declarations (= slots), IF branches that gate behaviour (= data splices), INSERT/UPDATE/DELETE statements (= what tables get written), and EXEC calls to downstream SPs.

The key insight These files are the system, in machine-readable form. We just need to parse them and join them. No LLM, no fuzzy matching, no guesses.
03 · The splice taxonomy

Three kinds of "if this, then that."

A "splice" is a branch off the canonical journey — the if-this-then-that logic that makes real business workflows messy. Splices come from three independent sources. Click Show example on each card to see what each looks like in a real file.

Kind 1 · UI splice

User clicks a sub-screen

Every LINK-typed task in the form template's <meta> Tasks header opens a sub-screen. User-initiated, usually optional.

PoCrtMain has nine UI splices: Terms, TCD, Schedule, Budget, Dropship, PR-Coverage, SO-Coverage, Quality, Notes.

Kind 2 · Data splice

A slot value forces a path

Inside the SP code, every IF branch gating on a parameter is a data splice. The parameter is the slot; the value triggers extra required slots downstream.

Not optional — once the user commits to that slot value, the path is forced.

Kind 3 · State splice

A named UI state

Each <state id="..."> in the screen's _State.xml declares conditions under which controls show or hide. A UI mirror of an underlying data condition.

PoCrtMain has four named states: project_details_on/off, state_on/off.

So why three kinds? And which one actually fires?

Fair question. They describe the same conditional behaviour from three different perspectives:

  • UI splice — the user's action (clicking a sub-screen link)
  • Data splice — the system's behaviour (which SP branch executes)
  • State splice — the screen's appearance (which controls become visible)

And here's the honest truth about which the bot actually uses at runtime:

Splice kind Source When the bot uses it Visible in trace?
UI .htm meta Tasks (LINK) Heavily used. When the user mentions a sub-screen ("custom payment terms", "5% discount", "ship to customer") the bot maps that to the right LINK task and walks the sub-screen. Yes — ui_splice_requested, splices_walked
Data SP IF branches Used for required-slot detection. When the user provides a slot value that triggers extra required slots (po_type=Capital → capital_proposal_id), the bot enforces those before committing. Yes — splice_triggered, additional_required_slots
State _State.xml <state> Mostly informational today. Extracted because it's in the artifact, but the bot doesn't currently consult these at runtime. They're a future visual-runtime hint. No (not surfaced in the bot's trace panel)
A useful way to think about it UI splice = the verb (the user clicks).  Data splice = the noun being checked (the slot value).  State splice = the consequence the user sees on screen. The first two drive bot behaviour. The third describes what would happen visually in the real Ramco UI.

The decorator contract

This is the principle that holds the architecture together:

The contract Captions are pulled deterministically from .htm attributes, attached as metadata, and read by the bot only when generating responses or updating the trace UI. Zero impact on which TRANS fires, which SPs run, which tables get written, which slots are required, or any other decision.

06 · Packaging for handoff

One model, many views.
Same data, repackaged for humans.

The bot reads one big POCRT.json — 36k lines, 1 MB, because it needs everything in one fast-loading file. But a human trying to review a journey can't tell signal from scaffolding. So we generate per-aspect "handoff views" — the same model split into focused files, plus a one-page Markdown summary.

The problem with the monolithic file

When you open POCRT.json in a browser, the canonical_spine (6 entries, ~80 lines) is followed by:

  • 273 splices (~150 KB)
  • 15 screens × dozens of slots each (~290 KB)
  • 222 SP chains (~640 KB)

It all blurs together. You can't tell at a glance: "what's the journey?", "what fires when?", "what gets written where?". The signal is buried.

The solution — one folder per activity

A new composer pass (c06_handoff_views.py) takes each monolithic journey JSON and writes a per-aspect folder. The bot's source of truth stays untouched; humans get tidy, focused files.

out/model/activities/POCRT/
├── 01_spine.json        4 KB    ← the canonical journey (6 entries, fits on a screen)
├── 02_splices.json      152 KB    ← UI + state + data splices, grouped
├── 03_screens.json      290 KB    ← form layout, slots, collect_steps
├── 04_sp_chains.json      640 KB    ← task → SP-chain mapping
├── 05_taxonomy.json       0 KB    ← what entity this journey produces / consumes
└── HANDOFF.md           4 KB    ← human-readable one-pager

What goes where

If someone asks…Hand them…
"What's the canonical journey for Create Direct PO?"01_spine.json — fits on one screen, 6 entries
"What conditional logic does this journey have?"02_splices.json — UI/state/data splices in their own buckets
"What's on each screen? How do the sections break down?"03_screens.json — including the new collect_steps
"Which stored procedures get called when?"04_sp_chains.json — task-keyed
"Does this journey produce or consume an entity?"05_taxonomy.json — under a kilobyte
"Give me a one-page summary I can show the team"HANDOFF.md — at-a-glance counts, spine table, commit chains, splice samples, screens table

The architectural rule

Same data, never duplicated The split views are derived from the monolithic file by c06_handoff_views.py. They're regenerated whenever the model regenerates. The bot still reads POCRT.json — these split files are presentation, not architecture. Editing them by hand would be pointless: they'd be overwritten on the next composer run.

This separation makes three things easier:

BenefitWhy
Stakeholder reviewEmail a single 4 KB HANDOFF.md instead of expecting them to scroll a 1 MB JSON
Targeted diffsWhen SPs change, only 04_sp_chains.json changes. When form sections change, only 03_screens.json changes. Easier code review.
Cross-journey overviewA future MODULE_HANDOFF.md can consolidate all 13 user-facing activities' HANDOFFs into one document

Across the module

Every user-facing PO activity now has its own folder. Spine sizes range from ~130 bytes (PoCopy, the smallest) to ~4 KB (PoAmnd, biggest — 10 commits). Every canonical spine fits on two screens.

07 · Cross-journey context

The bot's working memory.

When the user creates a PO and then says "now amend it", the bot needs to carry the PO number across journeys. Solved by a generic session memory driven by the entity taxonomy — zero PO-specific code.

The composer's c05_entity_taxonomy.py pass classifies every activity by parsing its description for verbs and nouns. "Create" → producer. "Amend" / "Approve" / "View" → consumer. The noun ("Purchase Order", "Purchase Request") names the entity kind.

ActivityRoleProducesConsumes
POCRTproducerPurchaseOrder · po_number—
POCRTQTNproducer + consumerPurchaseOrder · po_numberQuotation · quotation_no
POAMNDconsumer—PurchaseOrder · po_number
POAPPconsumer—PurchaseOrder · po_number
POVIWconsumer—PurchaseOrder · po_number

The same logic, unchanged, classifies any other Ramco module's activities. "Create" means produce — in PO, PR, GR, or PQ — the verb taxonomy is universal.

What happens at runtime

When the user issues a commit, the bot looks up the activity's taxonomy. If it produces an entity, the bot mints a synthetic ID (e.g. PO-NEW-001) and stores it in session memory keyed by entity kind.

When the user's next turn uses change-language (wait, actually, bump, change, modify) AND session memory has an entity of the right kind, the bot deterministically switches journey to the consumer journey and auto-fills the consumed-id slot from session memory. The user never types the PO number.

parse

The bot first calls its LLM classifier (Gemini 2.5 Pro). It gives the LLM only the user's message plus a fixed vocabulary of journey names and slot names from the model. The LLM is forbidden to invent new journeys or new slot names — it can only pick from the lists.

lookup
The bot opens the journey model for PoCrt

With journey = "PoCrt", the bot loads out/model/activities/POCRT.json. This file (built earlier by the composer) tells it everything about Create Direct PO: the canonical spine, the splices, the screens, the SP chains.

Step 3
detect splice
A data splice trips: po_type = Capital

The bot compares the user's slot values against the data-splice table loaded in Step 2. It finds a match: po_type == "Capital" is one of the known splice triggers for PoCrt. That triggers a rule: this commit cannot proceed unless capital_proposal_id is also provided.

The bot adds capital_proposal_id to its additional required slots list for this conversation.

splice detection:   splice_triggered: "po_type==Capital"   additional_required_slots: ["capital_proposal_id"]
Step 4
validate
Pre-commit validation BLOCKS the commit

The bot now wants to commit (wants_commit = true from Step 1). Before firing any TRANS task, it runs validation: does the slot state include every slot in additional required slots? It does not — capital_proposal_id is missing.

validation result:   required slots check: capital_proposal_id → MISSING   decision: block commit   validation_error_detected: true
Step 5
fire
No TRANS fires. No SPs run. No tables written.

Because validation blocked the commit, nothing happens on the database side this turn. The bot's trans_invoked stays null. The SP chain stays untouched. This is the safety property: the bot will never fire an SP it knows would fail validation.

The bot generates a clarifying reply

09 · Five takeaways

What this project taught us.

"Documents are commentary on the system, not the system itself. The screens, service catalog, and SPs are the source of truth — and they're deterministically parseable."

"Per-activity naming conventions are unstable. Score by task description, not name. TRN4 in PoCrt means Create-and-Approve; in PoApp it means Get-Quote-Line-No — totally different."

"The eval architecture matters more than the bot architecture. Switching from per-turn metrics to system-effect grading revealed bot v3 was 4× better than the original metric suggested."

"Context carry across journeys must be a separate generic layer. Session memory keyed by entity taxonomy makes the bot module-agnostic."

"LLM is the natural-language layer, never the structure layer. Every commit goes through the deterministic SP chain. The LLM can talk; only the parsed model can act."

How the parser turns this into model data
★p03 parser walks the DOM. When it hits a <table type="section"> it captures the caption, stitle, and gridctrl attributes. Sections with non-empty captions and stitle="1" become candidate steps.
★Garbage caption filter. Some captions are technical IDs like "PoCrtMain_mainscreen_SECTION17". The parser detects these by pattern and excludes them; only clean English captions become steps.
★Mandatory detection via CSS class. The label cell uses class="labelsmandatoryleft" instead of "labelsleft" for required fields. The parser counts these per section to produce the mandatory_count.
★The composer joins it all. Each titled section + its slot labels + its mandatory count = one entry in the screen's new collect_steps array. The bot reads this at runtime for the trace UI and response prose. Decisions stay untouched.
